# i18n — Spanish & Portuguese Translation Design

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Add full Spanish (`es`) and Portuguese (`pt`) translations to the LFF website, covering all UI chrome and all page content, using `next-intl` and a static-export-compatible route structure.

**Architecture:** English pages stay at their existing root URLs (`/`, `/about`, etc.) via a `(en)` route group. Spanish and Portuguese pages live under locale-prefixed paths (`/es/`, `/pt/`). All translations are pre-generated at build time — no runtime fetching.

**Tech Stack:** Next.js 15 App Router, `output: 'export'`, `next-intl` v4, TypeScript.

---

## 1. Routing & URL Structure

### Route groups

```
app/
  (en)/
    layout.tsx              ← sets lang="en", loads en messages, wraps in IntlProvider
    page.tsx                → /
    about/page.tsx          → /about
    our-work/page.tsx       → /our-work
    impact/page.tsx         → /impact
    get-involved/page.tsx   → /get-involved
    donate/page.tsx         → /donate
    contact/page.tsx        → /contact
    projects/
      page.tsx              → /projects
      [slug]/page.tsx       → /projects/[slug]
  [locale]/
    layout.tsx              ← generateStaticParams: ['es','pt']; sets lang, loads messages
    page.tsx                → /es/  |  /pt/
    about/page.tsx          → /es/about  |  /pt/about
    our-work/page.tsx       → /es/our-work  |  /pt/our-work
    impact/page.tsx         → /es/impact  |  /pt/impact
    get-involved/page.tsx   → /es/get-involved  |  /pt/get-involved
    donate/page.tsx         → /es/donate  |  /pt/donate
    contact/page.tsx        → /es/contact  |  /pt/contact
    projects/
      page.tsx              → /es/projects  |  /pt/projects
      [slug]/page.tsx       → /es/projects/[slug]  |  /pt/projects/[slug]
  layout.tsx                ← root layout (unchanged — fonts, meta, body)
```

The existing `app/` files move into `app/(en)/`. The `[locale]` subtree is a copy that receives the locale param.

### URL examples

| English | Spanish | Portuguese |
|---|---|---|
| `/` | `/es/` | `/pt/` |
| `/about` | `/es/about` | `/pt/about` |
| `/projects/lion-census` | `/es/projects/lion-census` | `/pt/projects/lion-census` |

### Static params

- `app/[locale]/layout.tsx` → `generateStaticParams` returns `[{ locale: 'es' }, { locale: 'pt' }]`
- `app/[locale]/projects/[slug]/page.tsx` → `generateStaticParams` returns every `slug` for each locale
- `app/(en)/projects/[slug]/page.tsx` → `generateStaticParams` unchanged (English slugs only)

## 2. next-intl Setup

### Config file: `i18n/config.ts`

```ts
export const locales = ['en', 'es', 'pt'] as const;
export type Locale = (typeof locales)[number];
export const defaultLocale: Locale = 'en';
```

### Request config: `i18n/request.ts`

```ts
import { getRequestConfig } from 'next-intl/server';
import type { Locale } from './config';

export default getRequestConfig(async ({ locale }) => ({
  messages: (await import(`../messages/${locale}.json`)).default,
}));
```

### next.config.ts

```ts
import createNextIntlPlugin from 'next-intl/plugin';
const withNextIntl = createNextIntlPlugin('./i18n/request.ts');
export default withNextIntl({ output: 'export', images: { ... } });
```

### Locale layouts

Both `(en)/layout.tsx` and `[locale]/layout.tsx` call `setRequestLocale(locale)` at the top (next-intl static rendering requirement) and wrap children in `NextIntlClientProvider`.

## 3. Translation Files

### Structure

```
messages/
  en.json     ← UI strings, English
  es.json     ← UI strings, Spanish
  pt.json     ← UI strings, Portuguese

data/
  data.json       ← existing English content (unchanged)
  data.es.json    ← full Spanish content (same shape as data.json)
  data.pt.json    ← full Portuguese content (same shape as data.json)
```

### messages/*.json — UI strings only

All keys are namespaced. Example shape:

```json
{
  "nav": {
    "about": "About",
    "ourWork": "Our Work",
    "projects": "Projects",
    "impact": "Impact",
    "getInvolved": "Get Involved",
    "donate": "Donate"
  },
  "footer": {
    "tagline": "Wildlife conservation and community development.",
    "rights": "All rights reserved."
  },
  "projects": {
    "allProjects": "All Projects",
    "moreProjects": "More Projects",
    "fieldWork": "Field Work",
    "aboutThisProject": "About this project",
    "ourCommitment": "Our Commitment",
    "previous": "Previous",
    "next": "Next"
  },
  "common": {
    "learnMore": "Learn More",
    "donate": "Donate",
    "readMore": "Read More"
  }
}
```

### data/data.es.json and data/data.pt.json

Identical JSON shape to `data/data.json`. All text fields (titles, body, headlines, excerpts, button labels within blocks, project titles, excerpts) are translated. Non-text fields (slugs, image URLs, dates, IDs, categories) are copied verbatim from `data.json`.

### lib/content.ts and lib/projects.ts changes

Both files gain a `locale` parameter:

```ts
export function getPage(slug: string, locale: Locale = 'en'): Page | undefined {
  const data = locale === 'en' ? dataEn : locale === 'es' ? dataEs : dataPt;
  return data.pages.find(p => p.slug === slug);
}
```

English (`data.json`) is always the fallback — if a key is missing in a translated file, `getPage` falls back to English silently.

### Machine translation script

A one-time Node.js script (`scripts/translate.ts`) reads `data/data.json` and `messages/en.json`, calls the DeepL or Google Translate API (or a local LLM), and writes `data/data.es.json`, `data/data.pt.json`, `messages/es.json`, `messages/pt.json`. The script is not part of the build — it runs once, outputs are committed.

## 4. Language Switcher

### Component: `components/ui/LocaleSwitcher.tsx`

A `'use client'` component. Uses `useLocale()` from next-intl and Next.js `usePathname` + `useRouter` to navigate to the same path in the target locale.

```
EN · ES · PT
```

- Active locale: `font-semibold text-ink`
- Inactive locales: `text-ink/40 hover:text-ink`
- Separator: `text-line`

### Placement

- **Desktop navbar** — far right, between the last nav link and the Donate button
- **Mobile Sheet** — bottom of the menu list, above the Donate button

### Navigation logic

```ts
function switchLocale(target: Locale) {
  const currentPath = pathname; // e.g. /es/about or /about
  // strip current locale prefix if present
  const pathWithoutLocale = currentPath.replace(/^\/(es|pt)/, '') || '/';
  // build new path
  const newPath = target === 'en' ? pathWithoutLocale : `/${target}${pathWithoutLocale}`;
  router.push(newPath);
}
```

## 5. SEO

Each locale layout adds `hreflang` alternate links in `<head>`:

```html
<link rel="alternate" hreflang="en" href="https://theluigifootprints.org/about" />
<link rel="alternate" hreflang="es" href="https://theluigifootprints.org/es/about" />
<link rel="alternate" hreflang="pt" href="https://theluigifootprints.org/pt/about" />
<link rel="alternate" hreflang="x-default" href="https://theluigifootprints.org/about" />
```

`<html lang>` is set to the correct locale on every page.

Page `<title>` and `<meta description>` come from translated `data.*.json` SEO fields (same keys as existing `seo.title` / `seo.description`).

## 6. Font

Both Fraunces and Inter in `app/layout.tsx` add `'latin-ext'` to their `subsets` array, enabling correct rendering of `ã`, `ç`, `ñ`, `é`, `ó`, `ú` and other characters used in Spanish and Portuguese.

```ts
const fraunces = Fraunces({
  subsets: ['latin', 'latin-ext'],
  ...
});
const inter = Inter({
  subsets: ['latin', 'latin-ext'],
  ...
});
```

## 7. Implementation Order

1. Install `next-intl`, create `i18n/config.ts` and `i18n/request.ts`, update `next.config.ts`
2. Write `messages/en.json` with all UI strings (extracted by reading current components)
3. Run machine translation script → generate `messages/es.json`, `messages/pt.json`
4. Generate `data/data.es.json` and `data/data.pt.json` from `data/data.json`
5. Update `lib/content.ts` and `lib/projects.ts` to accept `locale`
6. Move `app/` pages into `app/(en)/`, create `app/[locale]/` mirror
7. Update both locale layouts (`(en)` and `[locale]`) to use next-intl
8. Build `LocaleSwitcher` component, wire into Navbar and mobile Sheet
9. Add `hreflang` tags to both locale layouts
10. Add `latin-ext` font subsets
11. Build verification — confirm all 3 × all pages emit correctly
